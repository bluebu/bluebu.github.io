前四篇讲了框架、TDD、DDR、效率守卫——都是机制层面的东西。但如果你想在自己的项目里落地 Agent Harness，第一个要回答的问题其实是：怎么拆？哪些活儿该交给哪个 Agent？边界画在哪里？

<!--more-->

## 先说一个踩过的坑

一开始我们只有一个 Agent，什么都干。改语法、写 Java、写 Python、跑测试、更新文档——全在一个会话里完成。看起来效率很高，实际上问题不断。

最典型的就是越权。让它改 Java 实现，它“顺便”把测试文件也改了。让它加个 IR 节点，它“顺便”把三种语言的实现都写了——写得还不一定对。一个什么都能碰的 Agent，就像一个什么权限都有的数据库账号，迟早出事。

做金融系统设计有个基本原则：最小权限。每个模块只拿到它完成工作所需要的最少权限，多一分都不给。Subagent 的拆分逻辑本质上就是这个。

## 我们的 9 个 Agent

项目是一个 DSL，三种语言实现，外加 Playground 和文档系统。最终拆成 9 个：

```
Agent                模型      职责
───────────────────────────────────────────────────────
DSL 专家             opus      语法设计（IR、ANTLR4、版本号）
测试工程师           sonnet    一致性测试（CSV 用例设计、边界覆盖）
Java 实现专家        sonnet    Java 运行时实现
TypeScript 实现专家  sonnet    TypeScript 运行时实现
Python 实现专家      sonnet    Python 运行时实现
Playground 工程师    sonnet    演示环境 + Docker 部署
UI 组件工程师        sonnet    编辑器、表格组件包
文档专家             sonnet    MkDocs 文档站
配置审计专家         opus      审计 Agent Harness 配置（只读）
```

9 个里面只有两个用 opus：DSL 专家和配置审计专家。前者做语法设计，是整个系统最核心的架构决策；后者做配置审计，需要深度交叉推理。剩下都用 sonnet，够用就行。跟做系统架构选型一个道理——不是所有服务都需要跑在最贵的机器上。

## 拆分的三条原则

回头看，有三条原则是反复验证过的。

**一个文件只归一个 Agent 写** — 铁律。`test/runtime/` 下的 CSV 只有测试工程师能写，实现 Agent 只能读。`grammar/wen.ir` 只有 DSL 专家能改。如果两个 Agent 都能改同一个文件，职责边界就是模糊的，迟早打架。这跟数据库设计里的“单一写入者”原则一模一样——多写者意味着锁、冲突解决、事务，能避免就避免。

**按产出物拆，不按技能拆** — 一开始想过按“前端/后端/测试”这种技能维度拆，后来发现不好使。比如 Playground 的后端有 Java、Python、TypeScript 三个版本，如果按语言拆，Playground 工程师的活就散到三个 Agent 里了。最终是按“这个 Agent 交付什么”来拆——Playground 工程师交付一个能跑的演示环境，不管它里面有几种语言。

**贵的模型用在刀刃上** — 模型选择不是越贵越好，是匹配任务复杂度。语法设计需要在多种可能性之间做权衡，用 opus；实现类工作有明确的输入输出和验收标准，sonnet 足够。

## 文件权限表

每个 Agent 的定义文件里都有一张权限表，以 Java 实现专家为例：

```
路径                                权限    说明
───────────────────────────────────────────────────────────
impl/java/src/.../runtime/          读写    运行时实现，主战场
impl/java/src/.../generated/        只读    代码生成产物，碰了就废
impl/java/src/test/                 读写    Java 单元测试
test/runtime/                       只读    一致性测试 CSV，只看不改
grammar/wen.ir                      只读    IR 定义，参考设计意图
impl/typescript/                    ──      不可见
impl/python/                        ──      不可见
```

这张表不是建议，是硬约束。Agent 试图修改没有写权限的文件会被拦住。

有个容易忽略的点：**只读也要显式声明**。不是“没列出来的都能读”，而是明确告诉 Agent“这些文件你需要参考，但不能改”。这样它知道去哪里找上下文，同时知道边界在哪。而“不可见”意味着这个 Agent 完全不知道那些文件的存在——Java 实现专家不需要看 Python 的代码，看了反而容易受干扰。

## Agent 之间怎么协作

9 个 Agent 不是各干各的，它们之间有明确的协作链路。以“实现一个新语法”为例：

```
DSL-Expert ──→ Test-Engineer ──→ Java ──→ TypeScript ──→ Python ──→ Playground ──→ Doc
    │               │              │          │             │
    │  design IR +  │  design      │  TDD     │  TDD        │  TDD
    │  grammar      │  test cases  │  strict  │  strict     │  strict
    │               │              │          │             │
    └─── plan ──────┴─── plan ─────┴── plan ──┴─── plan ───┘
```

关键设计：**Agent 之间不直接对话**。前序 Agent 的产出写进 plan 文件，后序 Agent 从 plan 文件读取输入。跟消息队列的思路一样——生产者和消费者解耦，中间靠持久化消息传递。

好处有两个。一是可审计——每个 Agent 做了什么决策、基于什么输入，plan 文件里都有记录，出了问题翻一下就知道是哪个环节。二是容错——某个 Agent 的工作需要重做，不用从头跑整条链路，从对应的 plan 文件接上就行。

## 容易混淆的边界

有几个边界一开始没想清楚，吃过亏：

**UI 工程师 vs Playground 工程师** — 都跟前端有关，但职责完全不同。UI 工程师管 `packages/` 下的独立组件包（编辑器、表格），Playground 工程师管 `toolkit/playground/` 下的演示应用。组件包是被集成的，Playground 是集成者。让同一个 Agent 管，它会搞混“改组件 API”和“改 Playground 用法”的区别。

**测试工程师 vs 实现专家** — 测试工程师写测试，实现专家写实现，这个清楚。容易出问题的是：测试跑不过的时候谁来判断是测试错了还是实现错了？答案是实现专家先假设测试是对的，去修实现。如果确认是测试本身有问题，上报给流程编排者，由测试工程师来改。实现专家绝对不能自己改测试。

**DSL 专家 vs 实现专家** — DSL 专家输出设计方案（IR 节点定义、语法规则变更），但不写实现代码。实现专家拿着设计方案写代码，但不改语法定义。设计和实现分离，任何工程体系里的基本功。

## 怎么写一个 Agent 定义

一个 Agent 定义文件的骨架：

```markdown
---
description: 一句话说明职责
model: opus / sonnet
tools: 可用的工具列表
---

## 角色定义
你是谁，负责什么。

## 核心文件
每次工作前必须先读的文件列表。

## 文件权限
路径 / 权限 / 说明。

## 工作流程
按场景分节，每个场景列步骤 + 验收命令。

## 命令速查
常用的 make / shell 命令。

## 规范约束
硬性红线，不能违反。
```

几个要点：

**“核心文件”容易被忽略。** 它定义的是 Agent 启动时必须先读的文件——不是等调用方在 prompt 里告诉它去读什么，而是 Agent 自己知道“我每次干活之前得先看看这几个文件”。这是让 Agent 具备自主判断能力的关键。

**“工作流程”要按场景拆。** 同一个 Agent 在不同场景下步骤不同。比如 Java 实现专家在“新功能”和“Bug 修复”下 TDD 模式不一样（strict vs fix），验收标准也不一样。不要写一个笼统流程企图覆盖所有场景。

**“规范约束”要写具体禁止项。** “保持代码质量”是废话，"不修改 `generated/` 目录下的文件"是规则。

## 拆分的起手式

如果你要在自己的项目里设计 Subagent，从三个问题入手：

> 项目有哪些**独立的产出物**？每个产出物大概率对应一个 Agent。
>
> 哪些文件**绝对不能被错误修改**？这些文件的写权限要严格控制。
>
> 哪些决策**需要深度推理**，哪些是**常规执行**？前者用好模型，后者用经济模型。

拆完之后画一张权限矩阵，确认没有两个 Agent 对同一个文件都有写权限。如果有，说明边界没画清楚，回去重新拆。
