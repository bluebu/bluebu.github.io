[上一篇](/posts/agent-harness-subagent)讲了怎么拆 Agent、怎么画权限边界。但 Agent 拆好了只是有了一支团队，你还需要一套剧本告诉他们怎么配合。这个剧本在我们的体系里叫 Skill。

<!--more-->

## Skill 是什么

Skill 就是一个工作流模板。它定义了一个完整任务从需求收集到交付的所有步骤，包括每一步委派哪个 Agent、输入输出是什么、用什么方式确认。

举个例子，`/syntax` 这个 Skill 定义了“实现一个新的 DSL 语法”的完整流程，从需求理解到三语言实现再到文档更新，一共 12 个步骤，串了 7 个 Agent。你只需要敲一个命令，整条流水线就转起来了。

但 Skill 不只是步骤列表。它还包含两个关键协议：**Intake**（开工前怎么收集需求）和 **Review**（过程中怎么确认）。这两个协议决定了人和 AI 之间的协作节奏。

## Intake：开工之前先把需求收齐

做过项目管理的人都知道，最大的浪费不是写错代码，是需求没对齐就开工。改代码的成本远低于重做的成本。

Intake 协议要求每个 Skill 在正式开工之前，先用一张字段表收集完整需求。以 `/syntax` 为例：

```
字段              必须    推断规则
──────────────────────────────────────────────
Jira 票号         是      --
语法功能描述      是      --
作用域            是      从功能描述推断
功能测试 case     是      从功能描述推断
```

收集流程分三步：

> **字段收集** — 逐项检查用户输入。必须字段缺失时，有推断规则就推断，没有就问。
>
> **需求确认** — 所有字段填完后展示给用户确认。推断出来的值会标注出来，让你知道哪些是 AI 猜的。
>
> **边界追问** — 确认后，针对可能有歧义的边界行为做最后一轮澄清。

关键规则是：信息完备就不多问。如果用户一条消息把所有字段都说清楚了，直接进确认环节，不搞一问一答的确认循环。这条规则听起来简单，但不写明的话 AI 真的会逐个字段跟你确认一遍——效率守卫里“确认循环”这个反模式，很多时候就是 Intake 没设计好导致的。

不同的 Skill 字段表差异很大。`/push`（发版）只需要一个版本号，`/ui`（组件开发）需要目标组件、任务类型、验收标准，涉及样式的还必须提供参考截图。字段表越精确，后面跑偏的概率越低。

## Review：三种确认节奏

Skill 执行过程中，人需要介入到什么程度？不同的任务答案不一样。Review 协议定义了三种模式：

**step** — 每步产出都展示给用户，等确认了才进下一步。适合复杂任务，比如 `/syntax` 涉及语法设计、三语言实现、文档更新，任何一步出问题后面都白做，所以每步都要确认。

**final** — 中间步骤自动推进，只在最终执行不可逆操作前确认。适合 `/ui` 这种任务——组件开发过程中不需要逐步确认，但最终提交前看一眼。

**none** — 全程自动，只展示最终结果。适合 `/push` 这种标准化流程——changelog、版本号、commit、squash、push、创建 MR，每一步都有明确的规则，不需要人介入。

选择的依据很简单：**不可逆操作越多，确认粒度越细**。语法设计是不可逆的（改了 IR 后面的实现都要跟着变），所以 step。组件代码随时能改，所以 final。发版流程每一步都有明确的回退方案，所以 none。

这跟做交易系统的审批流设计是一个道理。大额交易需要逐级审批，小额交易自动放行，系统内部的技术操作只需要事后审计。不是所有操作都值得同等程度的关注。

## 实际的几个 Skill

看几个具体的例子，从简单到复杂。

### /push — 全自动发版

最简单的 Skill，Review 模式 none，Intake 只收一个版本号。流程固定：

```
/changelog  →  /version  →  git commit  →  /squash  →  git push  →  PR
```

每一步都委派给对应的 Command（比 Skill 更小的执行单元），失败了自动停止。PR 那一步做了幂等设计：没有 MR 就创建，已有 MR 就更新，已合并就询问。不管你跑几次，结果都是确定的。

这种全自动 Skill 的关键是**每一步都有明确的成功/失败判定**。不存在模糊地带，不需要人来判断“这步算做完了吗”。

### /ui — 半自动组件开发

Review 模式 final，Intake 字段比较多：

```
字段          必须    推断规则
──────────────────────────────────────────────
目标组件      是      从描述推断（WenEditor / WenTable）
功能描述      是      --
任务类型      是      从描述推断（新增 / 修复 / 样式 / 行为变更）
影响范围      是      从目标组件推断
验收标准      是      从功能描述推断
参考样式      否      涉及 UI 样式时必须提供
```

注意“参考样式”这个字段。它平时不是必须的，但涉及样式调整时变成必须。这是从效率守卫的“美化漩涡”反模式里学到的教训——没有参考就改样式，一定会陷入“不好看→改→还是不好看→改回去”的死循环。Intake 在这里起到的作用是**把问题拦在开工之前**。

流程很短，就两步：委派 ui-engineer 实现，需要的话再委派 doc-writer 更新文档。中间自动推进，最终提交前确认。

### /syntax — 最复杂的编排

Review 模式 step，12 个步骤，串 7 个 Agent。分两个阶段：

**设计阶段** — 不写代码，只做决策：

```
步骤    委派            产出                   写入 plan 章节
──────────────────────────────────────────────────────────────
 0      --              获取 Jira，创建 plan    需求描述 + Jira 背景
 1      dsl-expert      需求分析                需求分析
 2      test-engineer   测试用例设计            测试用例
 3      dsl-expert      IR + 语法规则设计       IR 设计 + 语法规则 + 实现计划
 4      --              创建 Git 分支           --
```

每一步的产出都写进 plan 文件对应的章节。设计阶段结束后，plan 文件就是一份完整的实施方案，后面的实现 Agent 读这份方案就知道要做什么。

**实现阶段** — TDD strict，逐语言闭环：

```
步骤    委派                  验收命令
──────────────────────────────────────────────────
 5      impl-java             make java
 6      impl-typescript       make typescript
 7      impl-python           make python
 8      test-engineer         make test
 9      playground-engineer   make playground_build
10      doc-writer            make doc
11      --                    /squash + push + MR
```

这里能看到前面几篇讲的东西是怎么组合在一起的：TDD 红绿灯约束每种语言的实现节奏，单语言闭环保证逐个通过，step 模式让人在每种语言完成后确认一次，plan 文件在 Agent 之间传递上下文。

## 端到端的流程

三个 Skill 可以串成一条完整的交付链路：

```
/design          设计方案，创建 Jira ticket
    ↓
/syntax CF-xxxx  基于 ticket 实现语法（设计 + 三语言 + 文档）
    ↓
/push            changelog → 版本号 → commit → push → MR
```

从一个想法到一个可发布的版本，中间的每一步都有明确的 Agent 负责、有具体的验收标准、有对应的确认机制。人需要做的就是在关键节点确认方向没错，剩下的交给流水线。

## 写一个 Skill 的要点

如果你想在自己的项目里设计 Skill，几个经验：

**先定 Intake 字段表**。这决定了需求收集的质量。字段越精确，后面跑偏的概率越低。推断规则要写清楚——哪些信息可以从上下文推断出来，哪些必须用户明确提供。

**Review 模式匹配风险等级**。不要所有 Skill 都用 step，也不要都用 none。关键是看操作的可逆性——能轻松撤销的步骤自动推进，不可逆的步骤等人确认。

**每步都要有明确的产出和验收标准**。“实现 Java 部分”太模糊，"委派 impl-java，验收命令 `make java`"才是可执行的。

**复杂 Skill 用 plan 文件串联**。设计阶段的产出写进 plan，实现阶段从 plan 读取输入。这样即使中间断了（会话超时、需要排期），随时可以接上。
