前六篇把 Agent Harness 的各个机制都拆开讲了一遍。但如果你现在打开自己的项目准备落地，大概率会懵——9 个 Agent、54 条规则、6 个 Skill，这些是迭代了几个月的结果，不是第一天就该有的东西。

这篇聊怎么从零开始，一步一步搭起来。

<!--more-->

## 第一天只需要三样东西

别被前面文章里的复杂度吓到。我们第一天上线 Agent Harness 的时候，全部配置加起来就三样：

**一个 CLAUDE.md** — 写清楚项目是干什么的、目录结构长什么样、构建命令是什么。不用写规则，不用写约束，就把项目的“说明书”写好。这是所有后续工作的基础——AI 至少得知道自己在跟什么项目打交道。

**一个 Agent** — 对，就一个。不用拆。先让 AI 在一个会话里完整地跑一遍你的核心工作流。跑的过程中你自然会发现它在哪里跑偏、在哪里越权、在哪里犯蠢。这些发现比你提前设计任何架构都有价值。

**一条 TDD 规则** — 如果你的项目有测试，加一条最简单的规则：改代码之前先跑测试确认现状，改完之后再跑一遍确认没坏。就这一条，效果立竿见影。

第一天的目标不是“搭好体系”，是“开始用，开始发现问题”。

## 什么时候拆第二个 Agent

出现越权的时候。

我们的经验是，第一个触发信号通常是 AI 改了不该改的文件。让它改实现，它把测试也改了。让它加功能，它把不相关的代码也重构了。

这时候你拆出来的不是按技能分的“前端 Agent”和“后端 Agent”，而是按“谁能碰什么文件”分的。测试文件归测试 Agent，实现文件归实现 Agent，两者互不触碰。

具体的判断标准很简单：**当你第二次对 AI 说“别动那个文件”的时候，就该拆了**。第一次可能是意外，第二次说明这个边界需要硬约束，靠嘴说是记不住的。

不要提前拆。你觉得“将来可能需要”的 Agent，大概率不需要。等问题真的出现再拆，拆出来的边界才是准确的。

## 什么时候加效率守卫

同一个坑踩了三次的时候。

效率守卫不是一开始就该有的东西。它是从实际痛点里长出来的，不是凭空设计出来的。

我们的第一条效率守卫规则是“同一个文件修改三次以上就暂停”。因为真的遇到了这个问题——和 AI 反复改同一个文件的样式，改了五轮，最后改回了第二轮的状态。浪费了一个小时。

第二条是“连续五轮对话没产出代码就暂停”。因为真的遇到了——跟 AI 讨论一个语法设计讨论了八轮，最后发现其实第三轮的方案就是最终方案，后面五轮全是无效的细节纠结。

每条守卫规则背后都对应一次实际的效率损失。如果你还没踩过某个坑，不用急着加对应的守卫。加了也不知道阈值该设多少，太紧打断太多，太松形同虚设。

## 规则怎么迭代

靠数据，不靠感觉。

每两周跑一次效率回顾（我们的 `/review-me`）。看最近两周的工作记录，找到效率最低的三个时段，分析原因。如果原因指向一个缺失的规则，加上。如果指向一条规则的阈值不对，调整。如果指向一个 Agent 的权限太宽，收紧。

不要凭感觉加规则。“我觉得应该加一条 XX 规则”——先看数据，过去两周这个问题出现过几次？如果一次都没有，不用加。规则是有成本的，每多一条规则就多占一点上下文窗口，多一点 AI 需要遵守的约束。规则膨胀本身就是一种效率损失。

也不要轻易删规则。如果一条规则两个月没被触发过，先确认是规则生效了所以问题消失了，还是规则本身条件写得不对从来没匹配过。前者说明规则有用，别删；后者说明规则该修。

## Skill 从最痛的流程开始

不要第一个 Skill 就搞最复杂的端到端流程。先找团队里最痛的一个重复操作，把它自动化。

我们的第一个 Skill 是 `/push`——发版。因为每次发版都是同样的步骤：生成 changelog、更新版本号、commit、squash、push、创建 MR。手动做容易漏步骤，而且每次都要记一堆命令。自动化之后直接一条命令搞定，Review 模式 none，全程不需要人介入。

第一个 Skill 建议选 Review 模式 none 的——全自动流程更容易验证对不对，因为结果是确定性的。Review 模式 step 的 Skill 更复杂，每步都需要人确认的逻辑需要更多调试。

有了第一个 Skill 之后，团队会建立信心：“哦，这东西真能省事。”然后自然会有人问“XX 流程能不能也自动化”，第二个、第三个 Skill 就顺理成章了。

## 几个常见的搭建误区

**规则太多太早** — 第一天写了 20 条规则。AI 记不住，你也记不住哪条规则在什么场景下生效。先从 3-5 条核心规则开始，遇到问题再加。

**Agent 拆太细** — 三个人的团队拆了 15 个 Agent。维护 Agent 定义文件本身就变成了负担。Agent 数量跟项目复杂度成正比，不是越多越好。我们的项目三种语言、Playground、文档系统、组件包，才拆了 9 个。一个普通的后端项目，3-4 个 Agent 可能就够了。

**Review 全用 step** — 每一步都等人确认。确认本身是有成本的，每次确认都打断你的工作节奏。对于确定性高的步骤，用 final 或 none。step 只用在真正需要人判断的环节。

**Intake 字段太多** — 开工之前问了十个问题。用户的耐心是有限的，问太多他会开始随便回答，效果反而不如不问。只收真正影响实现方向的关键字段，能推断的就推断。

**不做回顾** — 搭好了就不管了。体系是要养的。规则会过时，Agent 边界会随着项目演进需要调整，Skill 流程会有优化空间。不定期回顾的 Agent Harness 会逐渐僵化，最终变成负担而不是助力。

## 一个参考时间线

我们的实际搭建过程，供参考：

```
第 1 周    CLAUDE.md + 1 个 Agent + TDD 规则
           开始用，开始记录问题

第 2 周    拆出第 2 个 Agent（测试 vs 实现）
           加了 Git 规范（禁止直推 master）
           第 1 个 Skill：/push（全自动发版）

第 3 周    拆出语言专属 Agent（Java / TS / Python）
           加了单语言闭环规则
           第一次效率回顾，加了两条效率守卫
           第 2 个 Skill：/syntax（端到端语法实现）
           Intake + Review 协议成型

第 4 周    DDR 模式落地（/review-me + /review-claude）
           9 Agent + 6 Skill + 50+ 规则
           体系基本稳定，进入微调阶段
```

一个月，从零到成型。节奏很快，但每一步都是被实际问题驱动的，不是提前规划好的。

## 最后

回顾这个系列，Agent Harness 的核心就一句话：**用架构约束 AI，而不是用 prompt 求 AI**。

prompt 是请求，架构是制度。请求依赖对方的配合意愿，制度不依赖。你不需要 AI 理解你的苦心，你只需要它在你画好的框里做事。这跟设计任何系统都一样——好的架构让做对的事情变得容易，让做错的事情变得困难。

这套东西不神秘，也不复杂。它就是传统工程管理经验在 AI 协作场景下的新应用。如果你做过系统架构、带过团队、搞过流程优化，这些思路你都见过，只是换了个对象。

开始用就好了，别想太多。
