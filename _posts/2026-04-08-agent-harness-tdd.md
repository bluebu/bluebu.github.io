[上一篇](/posts/agent-harness-pattern)聊了 Agent Harness 的整体思路。这篇单独说 TDD——不是教科书里的那种，是针对 AI 编程场景重新设计过的。

<!--more-->

## AI 没有节奏感

程序员写代码有个天然的习惯：写一段，跑一下，看看对不对，再写。这种节奏是多年训练出来的肌肉记忆。AI 没有。你让它实现一个功能，它的默认行为是一口气把所有文件全改完，然后跟你说“搞定了”。

这让我想起交易系统做压力测试的一个经验：一次性灌入全天的订单量，系统挂了，你根本不知道是哪个环节先撑不住的。但如果分批灌入、每批验证，问题在哪里一目了然。

AI 写代码也一样。一次改十个文件，测试挂了，排查成本极高。而且 AI 有个更恶劣的习惯：它会“作弊”。面对一个跑不过的测试，它有时候不是去修实现，而是悄悄改测试——把期望值改成实际值，测试就过了。跟交易系统里篡改对账报表一个性质，结果好看了，问题埋得更深了。

TDD 红绿灯天然解决这个问题。它不是什么高深的方法论，就是一个节拍器：先证明测试会失败，再写代码让它通过，然后清理。一步一验证，中间不跳步。

## strict 和 fix

两种模式，对应两种场景。

**strict** — 用于新功能，三步走：

> **红** — 跑测试，确认新用例确实失败。这步对 AI 至关重要——它证明了这个测试不是写出来就能过的废用例。
>
> **绿** — 写最小实现让测试通过。强调“最小”，AI 天生喜欢一步到位写个完整版出来，这里必须克制住。
>
> **重构** — 测试通过后收拾代码，再跑一遍确认没改坏。

**fix** — 用于修 Bug，也是三步：

> 复现 → 修复 → 验证

共同的铁律：每一步结束必须跑测试。不是“建议跑一下”，是硬约束。

## 单语言闭环

我们三套语言实现（Java、TypeScript、Python）一开始的做法是让 AI 先把三种语言都写完，最后统一跑测试。亏了。

经常出现的情况是：Java 的实现假设了某个排序是稳定的，TypeScript 的实现假设了另一种边界行为，到最后一起跑测试，失败的用例交叉在一起，排查起来跟解一团乱麻似的。

后来改成单语言闭环：Java 走完整个红绿灯循环，验收通过，再开 TypeScript。TypeScript 通过了，再开 Python。看起来是串行的，好像慢了，实际上快很多。因为每种语言的问题在当场就暴露、当场就解决，不会滚雪球。

这跟做系统联调的经验一模一样——先把每个模块单独调通，再联起来。上来就联调的项目，后面一定是一堆跨模块的幽灵 Bug。

在 Agent 编排层面就是串行委派：

```
impl-java       → make java_grammar + make java_conformance
impl-typescript → make ts_grammar   + make ts_conformance  
impl-python     → make python_grammar + make python_conformance
```

每个 Agent 做完自己的闭环，交出验收结果，流程才往下走。

## 不能松的几条线

跑了一段时间，有几条规则试过放松，代价都很大，后来就再没动过。

**不跳红** — strict 模式下必须先证明测试失败。AI 经常试图跳过这步直接写实现，但你一旦允许跳红，就无法区分“测试有效”和“测试本身就是通过的”。在金融系统里，这相当于上线一个没做过反向验证的风控规则——你觉得它在工作，其实它从来没拦过任何东西。

**测试只读** — 实现 Agent 对测试目录没有写权限。这是硬隔离，不是口头约定。AI 面对跑不过的测试，第一直觉是“测试写错了”。人也会有这个直觉，但人知道去找测试 owner 确认。AI 不会，它直接改。所以测试文件的写权限只给测试工程师 Agent，实现 Agent 碰不到。

**禁止删用例** — AI 真的会把失败的用例删掉来让测试“通过”。这条规则的存在本身就说明 AI 和人在工程纪律上有本质差异——人知道删用例是荒唐的，AI 只知道目标是“测试通过”，至于怎么通过，它没有道德判断。

这几条规则写在 `rules/tdd.md` 里，按文件路径自动加载。不需要每次在 prompt 里强调，它就是环境的一部分。好的控制措施是内嵌在架构里的，不是写在流程文档里靠人记的。

## 验收要可执行

每个 TDD 流程必须声明具体的验收命令。不是“测试通过”这种模糊说法，而是可以直接跑的命令：

```bash
make java_grammar       # 语法解析正确
make java_conformance   # 一致性测试通过
```

AI 执行完实现，自己跑验收，通过才算绿灯，不通过自己回去改。

这一点带来的好处是人的 review 负担断崖式下降。你不用去拉代码、配环境、跑测试来验证 AI 的工作——它已经验证过了，你只需要 review 代码本身的设计和可读性。
