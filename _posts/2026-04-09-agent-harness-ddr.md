## 测试管不到的地方

TDD 守住了代码质量的底线，但有一类问题它管不到。

Agent 的文件权限表是不是设宽了？某条效率守卫规则是不是阈值不对，太灵敏一直打断人，或者太迟钝根本没用？某个 Skill 的流程步骤是不是可以精简？这些问题不会导致测试失败，不会触发 CI 报错，但长期积累下来，团队效率会一点一点被侵蚀。

做过金融系统运维的人对这类问题应该不陌生。交易系统跑着没问题，每天对账也平，但中间某个环节的延迟在悄悄增长，某个队列的积压在缓慢攀升。不定期做健康检查，等到出事的时候往往已经是系统性的了。

这类问题需要定期回顾。而回顾这件事本身也可以交给 AI——只要你给它一个清晰的框架，别让它漫无目的地“看看有什么问题”。

## Detect → Diagnose → Remediate

DDR 就是这个框架。名字是从风控领域借来的，那边叫“发现-评估-处置”，换了个皮而已。三个阶段：

**Detect（发现）** — 纯采集。扫描配置文件，比对规范，输出事实清单。不分析、不建议、不动手。比如审计 `.claude/` 配置时，Detect 阶段检查的是：

> Agent 定义有没有缺必填字段？规则文件的 paths 声明和实际目录结构对不对得上？有没有规则写了但从没被任何 Agent 加载过？

关键是只输出事实。“某 Agent 定义缺少 model 字段”是事实，“应该补上 sonnet”是建议——后者属于下一步的事。把采集和判断分开，是为了让 Detect 的结果可信。你不需要质疑它的动机，因为它没有动机，只是在做模式匹配。

**Diagnose（诊断）** — 拿到事实清单，分析根因，给出改进方案。但输出的不是代码变更，是一份 plan 文件。plan 里写清楚每个问题的现状、成因、建议改法。这步结束后要人确认。

**Remediate（修复）** — 人确认 plan 之后才执行。逐项修改，每改完一项标记状态。

三个阶段可以分开跑，也可以串起来一次走完。分开跑适合你只想看看现状但还没想好要不要改的时候。

## 为什么中间要有个 plan

这是整个 DDR 里最重要的设计决策。为什么不让 AI 发现问题直接改？

做过风控的人都理解：检查和处置必须分权。让同一个角色既发现问题又修复问题，等于没有检查。在金融行业这叫“前中后台分离”，道理一样。

具体到工程实践上：

**风险等级不同** — 看一眼有什么问题是零成本的，改配置可能影响所有人的工作流。这两件事不应该在同一次操作里无缝衔接，中间必须有个人工确认的卡点。

**可审查** — plan 落在 `.claude/plans/` 目录下，用 ticket 号命名，进了版本控制。任何人打开就能看到 AI 打算做什么、为什么。不同意某项建议，从 plan 里删掉，Remediate 阶段就不会执行它。

**可跨会话** — 今天跑 Detect + Diagnose 生成了 plan，明天再来跑 Remediate。AI 不需要重新理解上下文，plan 里都有。审计结果出来了但修改需要排期，这在实际操作中太常见了。

## 两个实际的 Skill

**`/review-claude`** — 审计 Agent Harness 的配置质量。说白了就是让 AI 检查自己的“操作系统”——Agent 定义写得对不对、规则文件有没有冲突、权限分配合不合理。听起来有点自我指涉的味道，但实际效果不错。AI 比人更擅长做这种交叉比对的脏活——某个 Agent 声明了某目录的写权限，但另一条规则里写着这个目录应该只读，这种不一致人很难发现，AI 很容易。

**`/review-me`** — 审计开发效率。分析最近一段时间的 Git log 和会话记录，识别低效模式。同一个文件被反复修改、某个流程的执行时间突然变长、某类任务的确认循环次数偏高——这些信号单独看不起眼，放在一起看往往能指向一个系统性的问题。

## DDR 和 TDD 的关系

两者管的是不同维度的事情。

用交易系统打比方就很清楚：

> TDD 是每笔交易的前置风控——下单前检查限额、检查合规、检查风险敞口。实时的，逐笔的。
>
> DDR 是季度的内部审计——回顾制度是否健全、流程是否有效、系统是否存在隐患。周期性的，全局的。

前者管住当下，后者管住趋势。组合起来，TDD 保证每次交付的代码是对的，DDR 保证产出代码的那套机制本身在持续优化。

## 写在后面

Agent Harness 这套东西说到底就是把传统工程管理的经验搬到了 AI 协作上：权限隔离、流程标准化、规则约束、定期审计。在金融 IT 领域，前中后台分离、STP、风控规则引擎、内审机制——每一项都有几十年的实践积累。AI 是新的，但管理 AI 的方法论不需要从零发明。

这套体系还在迭代，后面有新的实践再继续写。
