<!--more-->

## 一个数字：35 个版本，6 周

翻 CHANGELOG 会发现一个夸张的事实：从 v1.0.0 到 v1.1.1，一共 35 个版本，跨度大约 6 周。这些版本不是"改个 typo 刷版本号"——每个版本都有实质性的功能或架构变更：

- **语法层** — format、ternary、group_by filter mode 等新语法
- **架构层** — IR 重构（StreamFunc/CollectFunc 分类）、引擎并发安全改造、谓词下推优化
- **UI 层** — 编辑器和表格组件库从零到 npm 发布、Playground 和 Demo 应用
- **工程层** — 完整的 MkDocs 文档站、Docker 编排、CI/CD 流水线、450+ 一致性测试

这个速度在没有 AI 的时代是不可能的。不是因为写代码速度不够，而是因为**周边工作的体力成本太高**。

## "想做但懒得做"的清单

每个项目都有这么一份隐形清单——技术上知道该做，但评估下来工期太长、优先级排不上去，就一直搁着。我们这个项目的清单大概长这样：

**Playground（交互式试玩环境）** — 一个网页，上边写 DSL 脚本，下边看执行结果。还能切换 Java/TypeScript/Python 三个后端，对比结果是否一致。技术上不难，但前端工程量不小：Monaco Editor 集成、三语言后端 API、结果 diff 展示、示例库、语法参考面板、命令面板、深色/浅色主题切换……传统评估大概 2-3 周前端工作量。

**Test Runner（一致性测试运行器）** — Playground 内置的测试面板，前端同时调用 Java、TypeScript、Python 三个后端，跑 450+ 个 CSV 测试用例，逐条对比三语言的执行结果是否一致。不同环境、不同运行时，一个页面看全部结果。

**Demo 应用** — 比 Playground 更聚焦，展示编辑器和表格组件的集成方式。给其他团队看"怎么接入 DSL 引擎"的活文档。

**UI 组件库** — 把编辑器组件和结果表格组件从 Playground 里抽出来做成独立 npm 包，别的项目可以直接 `npm install` 用。涉及 Props 设计、事件协议、主题系统、文档、构建配置、发布流程。

**文档站** — MkDocs 驱动的完整文档，包括语法说明、引擎集成指南、组件 API、迁移指南、开发者指南。带自定义主题、语法高亮、交互式示例。

**三语言一致性测试框架** — 450+ 个 CSV 驱动的测试用例，每个用例包含脚本、输入数据、期望输出。三个语言各跑一遍，结果必须完全一致。

这些东西每一个单独拎出来都是"值得做但排不上优先级"的。全部做完？以前的评估是"至少三个月，得有专人"。

## AI 怎么改变了这个局面

直说结论：以上所有东西，大部分是 AI 辅助完成的。具体来说是 Claude Code 配合 Agent Harness 模式。

不是让 AI "自动写完整个项目"——那是幻觉。而是利用 AI 在几个特定环节的巨大效率优势：

### 体力活自动化

Codegen 模板、CSV 测试用例、CI 配置、Docker 编排、文档站的样板代码——这些工作的共同特点是**结构确定、重复度高、出错了容易发现**。AI 写这些东西的速度是人的 5-10 倍，而且不会因为写第 200 个测试用例时走神漏掉一个边界条件。

举个例子：450 个 CSV 测试用例的编写。每个用例是一行：脚本、输入、期望输出、描述。人写的话，前 50 个还能保持注意力，后面就开始复制粘贴改数字了。AI 写的话，你只需要说"给 group_by 的 filter mode 补充边界用例：空组、单行组、多个分组键"，它会系统性地枚举出来，比人想得全。

### 三语言平行实现

这是 AI 在这个项目里最大的价值点。我们的三语言实现有一个特点：**逻辑结构高度相似，但语言惯用写法不同**。Java 用 Stream API 做 filter/map/sort，TypeScript 用 Array 的链式调用，Python 用列表推导和 sorted()。

传统做法是先在一个语言（通常是 Java）里写好、调通，然后人工翻译到其他两个语言。这个"翻译"过程枯燥、容易出错，而且每次改动都要翻译三遍。

有了 AI，流程变成：Java 里写好逻辑 → 让 AI 参照 Java 实现生成 TypeScript 和 Python 版本 → 跑一致性测试 → 微调。"翻译"这一步从半天缩短到几分钟。更重要的是，AI 不会因为"翻译疲劳"在第三个语言里犯低级错误。

### 架构决策的快速验证：从"不敢动"到"随手重构"

IR 重构那次是个好例子。v1.0.18 把 `cc`、`sort`、`hd`、`highlight` 等函数从 CollectFunc 改分类为 StreamFunc。这个改动涉及三个语言的 IR 定义、AstBuilder、ExecutionVisitor、以及所有依赖 CollectFunc/StreamFunc 分类判断的逻辑。

以前这种"跨层重构"是最让人犹豫的——改动面大，回归测试量大，万一中间发现方向不对，回滚成本也大。所以经常的决策是"先凑合用着，等下个大版本再改"。技术债就是这么积累的：不是不知道该改，是**改不起**。

有了 AI 之后，同样的重构：让 AI 按新的分类规则改三个语言的实现，跑 450 个测试用例验证——连休息带思考，半小时搞定。方向不对？丢掉重来，也就再花半小时。

以前攒着的"等下个大版本再说"，现在变成了"现在就改，反正半小时的事"。**试错成本低了，架构决策就敢做了。**

### Playground 和 Demo：从"懒得做"到"顺手做了"

Playground 是最典型的"AI 让懒人项目落地"的例子。

以前觉得工期长是因为前端工程量大：Monaco Editor 的语法高亮配置、自动补全逻辑、三语言后端的 API 对接、结果展示表格、图表渲染、示例管理、深色模式、响应式布局……每一个都是"不难但费时间"的活。

用 AI 做的时候，这些变成了一系列明确的小任务。"用 Monaco Editor 做一个 DSL 脚本编辑器，支持语法高亮，这是 TextMate grammar 文件"——AI 生成一个能跑的版本。"加一个 Examples 面板，左边分类列表，点击加载到编辑器"——又一个能跑的版本。每个小任务 AI 都能快速产出一个 80% 完成度的结果，人只需要做 20% 的调优和集成。

编辑器和表格作为独立组件库发布也是类似的路径。Props 设计、事件协议定义、构建配置、npm 发布流程——这些"脚手架工作"AI 都能高效处理，人聚焦在核心的交互设计和边界用例上。

## Agent Harness：让 AI 不跑偏的关键

光有 AI 还不够。如果每次都是"打开对话窗口 → 描述需求 → 等 AI 输出 → 复制粘贴 → 手工调整"，效率提升有限，而且 AI 容易跑偏——它不知道你的代码规范、目录结构、构建流程、测试要求。

我们在这个项目里落地了 Agent Harness 模式，核心是四层约束：

**上下文管理** — 9 个 Subagent，每个有明确的文件权限范围。DSL 专家只能动 `grammar/` 和 `test/grammar/`，Java 实现专家只能动 `impl/java/` 和 Codegen 的 Java 模板。权限隔离让 AI 不会"好心"帮你改了不该改的文件。

**流程编排** — `/design` → `/syntax` → `/push` 的流水线。设计一个新语法时，`/design` Skill 产出结构化方案并自动创建 Jira ticket；`/syntax` 拿着 ticket 做三语言实现加测试；`/push` 一键完成 changelog → 版本号 → commit → squash → push → MR。每个环节该做什么是编排好的，AI 不用"创造性发挥"。

**护栏约束** — 效率守卫规则自动检测碎片化需求、美化循环等低效模式。TDD 规则要求"先写测试再写实现"。组件规范定义了编辑器/表格的通信协议和验证清单。

**反馈循环** — `/review-me` Skill 定期回顾工作记录，分析时间分布和效率瓶颈，产出优化建议。发现某类任务反复出错，就更新规则防止再犯。

这套体系让 AI 从"聪明但不可控的助手"变成"在明确边界内高效执行的队友"。35 个版本的高产出不是 AI 一个人的功劳，是**人定义边界、AI 在边界内高速执行**的结果。

## 改变的不只是速度

回头看，AI 对这个项目最大的改变不是"写代码快了"，而是**改变了什么值得做的判断标准**。

以前评估一个特性，心里有个隐含的公式：`价值 / 工期`。Playground 价值大但工期长，除以出来就是低优先级。UI 组件库对外有价值但对内开发成本高，排不上。

AI 把分母压缩了。以前三周的活现在三天能做，`价值 / 工期` 的比值直接翻了好几倍。于是那份"想做但懒得做"的清单，一项一项地变成了"做了也不费事"，然后就真做了。

文档站做了、Playground 做了、Demo 做了、组件库抽出来发了 npm 包、Docker 编排做了、450 个一致性测试写了、CI/CD 全链路打通了。这些东西放在一起，让一个"内部工具级别"的 DSL 项目，有了"正经产品"的样子。

这大概就是 AI 在工程项目里最实在的价值——不是替代人做决策，而是**让以前因为成本太高而搁置的好决策，终于能落地了**。
